## RAG: 题库 + 向量检索接入

### 背景
- 项目需要结合本地题库（可控）与 LLM 智能出题（灵活），形成 5+3 混合策略
- 题库需支持多岗位分类，且题目可能较长（多行描述）
- 检索需根据岗位名自动匹配题库，找不到时全库兜底

### 改动

#### 新增 RAG 模块
- 新建 `ai-service/app/rag/` 目录，包含：
  - `embedder.py`：通义 text-embedding-v4 封装，支持批量 + 重试
  - `vector_store.py`：Chroma 持久化客户端（本地 `./chroma_db`）
  - `loader.py`：题库解析（按空行切分，支持多行题目）
  - `retriever.py`：检索接口（岗位匹配 → 全库兜底）
  - `ingest.py`：CLI 入库脚本（`python -m app.rag.ingest [--reset]`）

#### 题库文件
- 创建 `ai-service/question_bank/` 目录，包含 5 份题库（每份 20 道，共 100 道）：
  - 前端工程师.txt、后端工程师.txt、AI产品经理.txt、AI应用开发.txt、全栈工程师.txt
- 格式：纯文本，空行分隔题目，支持多行题干；# 开头行视为注释

#### 配置项
- 追加 `.env`：
  ```
  EMBEDDING_MODEL=text-embedding-v4
  CHROMA_PERSIST_DIR=./chroma_db
  CHROMA_COLLECTION=interview_questions
  QUESTION_BANK_DIR=./question_bank
  RAG_TOP_K=5
  LLM_QUESTIONS_COUNT=3
  RAG_SCORE_THRESHOLD=0.25
  ```

#### 改造 `question.py`
- 新增 `generate_questions` 接口逻辑：
  1. 从 `jd_json.title` 匹配题库 category（子串匹配）
  2. 构造检索 query（JD 要点 + 简历技能）
  3. 检索 top-5 题库题（带相似度分数）
  4. 把已选题作为上下文，让 LLM 再生成 3 道互补题（聚焦差距点 + 项目深挖）
  5. 合并返回，每题带 `source: "rag" | "llm"` 字段
- 题库题 type 固定为 `knowledge_base`，LLM 题按策略标记 `project_deep_dive / job_competency / behavioral`

#### 依赖
- 追加 `chromadb==0.5.5` 到 `requirements.txt`
- `.gitignore` 排除 `chroma_db/` 持久化目录

### 收益
| 指标 | 改前 | 改后 |
|------|------|------|
| 出题来源 | 纯 LLM 8 道 | 5 道题库 + 3 道 LLM 互补 |
| 题库可维护性 | 无 | 纯文本文件，随时增删 |
| 岗位匹配 | 无 | 自动按 JD title 匹配题库 |
| 题目类型 | LLM 随机 | 题库题（知识）+ LLM 题（深挖/行为） |
| 响应时延 | 8 道串行 LLM | 检索本地向量库（毫秒级）+ 3 道 LLM |
| 成本 | 8 次 LLM 调用 | 3 次 LLM 调用（节省 62.5%） |

### 验证方式

1. **题库入库**：
   ```bash
   cd ai-service && venv\Scripts\python.exe -m app.rag.ingest --reset
   ```
   预期：解析 100 道题，耗时约 8s，collection 计数 100。

2. **接口测试**：
   ```bash
   curl -X POST http://localhost:8000/generate_questions \
     -H "Content-Type: application/json" \
     -d '{
       "resume_json": {"skills": ["React", "Node.js"]},
       "jd_json": {"title": "前端工程师", "requirements": ["React", "性能优化"]},
       "match_analysis": {"gaps": ["SSR 经验不足"]}
     }'
   ```
   预期返回 8 道题，其中 5 道 `source: "rag"`，3 道 `source: "llm"`，且 LLM 题聚焦 SSR 短板。

3. **配置可调**：
   - 修改 `.env` 的 `RAG_TOP_K` / `LLM_QUESTIONS_COUNT` 可动态调整比例
   - 题库文件增删后，重新运行 `ingest` 即可增量更新

## 🚨 关键修复：最后一题刚出，输入框就消失变成"结束面试"按钮

### 问题现象
用户答到倒数第二题（如 8 道题中的第 7 题）发送回答后，AI 流式输出第 8 题题干，
但**对话输入框立即被替换成「结束面试，查看报告」按钮**，导致用户无法回答最后一题。

### 根因
[`InterviewPage.tsx`](client/src/pages/InterviewPage.tsx) 输入区的渲染条件写错了：

```tsx
{currentIndex >= questions.length ? (结束按钮) : (输入框)}
```

`currentIndex` 表示「**当前正被问的题号**」（从 1 开始）：
- 第 7 题答完 → `streamChat(nextIndex=8)` → AI 流式输出第 8 题 → `setCurrentIndex(8)`
- 此时 `currentIndex(8) >= questions.length(8)` ✅ 命中，输入框被替换 ❌

但用户连第 8 题还没回答！正确语义是「**第 8 题已回答完**」时才结束，对应
`streamChat` 里的 `nextIndex > questions.length` 分支会触发 `setInterviewOver(true)`。
也就是说，此处的判断**比 `interviewOver` 早了一步**，造成误判。

### 修复
[`InterviewPage.tsx`](client/src/pages/InterviewPage.tsx) 输入区简化为：
- 删除嵌套的 `currentIndex >= questions.length` 三元判断
- 仅保留外层 `{!interviewOver && (...)}`，让结束流程**完全由 `interviewOver` 主导**
- 答完最后一题后，`streamChat` 中的 `nextIndex > questions.length` 分支负责把
  `interviewOver` 置为 true，聊天区原有的「面试圆满完成」结束卡片会自然显示

### 收益
| 指标 | 改前 | 改后 |
|------|------|------|
| 最后一题可回答 | ❌ 输入框直接消失 | ✅ 正常显示输入框 |
| 结束触发逻辑 | `currentIndex >= len` 与 `interviewOver` 双重判断，有错位 | 单一 `interviewOver` 主导，无歧义 |
| 跳题至最后一题 | 同样会提前显示按钮 | 同样修复 |

### 验证方式
1. 完整面试流程：答到最后一题应正常出现输入框，可输入回答；
2. 答完最后一题发送后，应看到 AI 收尾对话 + 「面试圆满完成」结束卡片 + 「查看评估报告」按钮；
3. 全程跳题至最后一题，最后一次跳题后应正常进入结束态。

### 备注
此 bug 暴露了一个常见的状态机设计反模式：**用同一个状态从不同维度推导 UI**，
当多个判断逻辑分散在不同地方（`streamChat` 用 `>`、UI 用 `>=`）时极易错位。
统一的修复思路是：**只让一个权威状态（`interviewOver`）控制 UI 分支**，所有其他状态只服务于业务流程推进。

## 🚨 关键修复：报告"已回答 0 题" + 面试结束后 AI 又弹欢迎语

### 现象
1. 候选人完整答完 8 道题，生成的报告中显示"已回答 0 题"，question_scores 为空数组；
2. 在结束界面或刷新后，AI 面试官又弹出一条："你好，欢迎参加今天的面试。请先简单介绍一下自己。"

### 根因分析（连锁 bug）

**根因 A：[`chat.py`](ai-service/app/api/chat.py) 边界条件硬编码欢迎语**
原逻辑：`if current_index > total or current_index <= 0: return 欢迎语`
当面试结束后前端某些时机会再次触达该接口（如最后一次出题完成后），由于 `current_index > total` 命中，AI 错误地返回了开场欢迎语，并被前端写入 `chat_history`。

**根因 B：欢迎语污染 chat_history → 报告 LLM 错乱**
被污染的 chat_history 末尾混入开场白后，原 [`report.py`](ai-service/app/api/report.py) 直接把 `chat_history` 整体序列化成 JSON 丢给 LLM，让模型自行从 `role/content` 推断"哪条对哪题"。LLM 在面对带噪历史时无法稳定建立题目-回答映射，最终输出空 `question_scores`，前端就显示"已回答 0 题"。

### 改动

#### 1. 修复 [`chat.py`](ai-service/app/api/chat.py) 边界
- 拆分两个独立分支：
  - `current_index <= 0`：返回欢迎开场白（首次进入）
  - `current_index > total`：返回结束语（"感谢参加本次面试..."），**不再误返欢迎语**
- 杜绝面试结束后再次触发欢迎语的可能。

#### 2. 重写 [`report.py`](ai-service/app/api/report.py) 评估生成
- 新增 `_is_valid_user_answer`：过滤空串、`[跳过]`、`skip` 等非实质回答；
- 新增 `_format_qa_pairs`：按题目顺序在 chat_history 中推进游标匹配用户回答，输出结构化纯文本：
  ```
  【题目1】xxx
  【候选人回答1】xxx
  ...
  【统计】共 N 道题，有效作答 M 道。
  ```
- 调用 LLM 时只传 `qa_pairs` + `total_questions`，**不再让 LLM 自己解析 chat_history**；
- 防御性兜底：若 LLM 仍返回空 `question_scores`，按题目数补充占位条目（score=0，注明"未能生成评估"），前端不会再出现"0 题"假象；
- 增加调试日志：`[Report] 生成开始：题目 N 道，有效回答 M 条 / 生成完成：question_scores=K 条`，便于排障。

#### 3. 同步 [`report_generation.py`](ai-service/app/prompts/report_generation.py) prompt
- 变量从 `{questions_json}` `{chat_history}` 改为 `{qa_pairs}` `{total_questions}`（与 report.py 保持一致，否则 LangChain 模板会抛 KeyError）；
- 新增三条**硬性要求**：必须逐题评分、未作答也要给出条目、`question_scores` 数组长度必须等于总题数；
- 评分标准明确化（1-5 分各档判定）。

### 收益对比

| 场景 | 修复前 | 修复后 |
| --- | --- | --- |
| 面试结束后边界触达 chat 接口 | AI 又弹欢迎语 | 返回结束语，不污染历史 |
| LLM 拿到的输入 | chat_history JSON 数组 | 结构化 `【题目X】→【回答X】` 配对文本 |
| LLM 输出 question_scores 为空 | 报告显示"已回答 0 题" | 按题目数补占位，附调试日志 |
| Prompt 变量 | 与 report.py 不一致（潜在 KeyError） | 完全对齐 |

### 验证方式
1. 启动 ai-service 和 server，前端完整答完一次面试（8 道题）；
2. 后端控制台应看到日志：`[Report] 生成开始：题目 8 道，有效回答 8 条`；
3. 报告页应显示 8 条逐题评分，且"已回答 X 题"等于实际答题数；
4. 面试结束后页面不应再出现 AI 的欢迎语开场白；
5. 异常场景：完全不答题直接结束，应直接返回提示性空报告（不调用 LLM）。

### 备注
本次双 bug 的核心教训：**LLM 不擅长在带噪数据中自行建立结构化映射**。将"对话历史 → 题目-回答配对"这一步在后端代码里做掉，远比塞给 LLM 自己推断更稳定。同时，prompt 模板与调用代码的变量名必须严格对齐，建议后续对所有 prompt 增加 unit test 检查变量集合一致性。

## 🚨 关键修复：报告内容空 + 重新面试时残留上一场的题目和回答

### 现象
1. 候选人答完所有题目，生成报告但**没有任何解析内容**（逐题点评、优势、改进建议等都是空的或只有占位文字）；
2. 面试结束后从 ReportPage 返回首页 → 重新上传简历 / JD → 进入新一场面试，AI 面试官**直接拿出上一场的题目和对话历史**，相当于"换了岗位但没换面试题"。

### 根因分析

**根因 A：handleEnd 清理后端会话用了 fire-and-forget**
[`InterviewPage.tsx`](client/src/pages/InterviewPage.tsx) 的 `handleEnd` 之前使用：
```ts
api.delete('/session/clear').catch(() => {});  // 不 await
navigate('/report');
```
这条请求在 navigate 之后才到 server。用户跳到 ReportPage 再回到 ResumeUploadPage / JdInputPage 时通常没事，但只要用户**立刻又开新一场面试**，`init()` 调 `/session/restore` 时旧的 Redis 会话很可能还在（删除请求尚未到达或已被打断），于是直接恢复了旧会话。

**根因 B：上游入口没有任何"开启新一场"的清理动作**
JdInputPage / ResumeUploadPage 没有清理 `/session/clear` 的逻辑——后端 session TTL 长达 2 小时，意味着只要在 2 小时内不结束面试就重新走一遍流程，必然踩到旧会话残留。

**根因 C：报告"无解析内容" 是 LLM 输出质量问题**
原 prompt 虽然要求逐题评分，但没有压实 comment 字段长度、风格、点评要求，导致 qwen-max 经常输出占位式空 comment 或缩水的 strengths/weaknesses。叠加之前的 question_scores 兜底逻辑（score=0 + "未能生成评估"占位），给用户的视觉就是**整页报告几乎全空**。

### 改动

#### 1. [`InterviewPage.tsx`](client/src/pages/InterviewPage.tsx) - handleEnd 同步清理
- `api.delete('/session/clear')` 改为 **await**，必须等服务端确认会话清除后再 navigate；
- 失败仅打 warn 不阻塞跳转（避免 Redis 异常导致用户卡在面试页）。

#### 2. [`JdInputPage.tsx`](client/src/pages/JdInputPage.tsx) - 新一场入口主动清理
- 用户提交新 JD 并完成 match 分析后，跳转 `/match-result` 之前同步执行：
  - `await api.delete('/session/clear')` 清掉后端 Redis 会话
  - `localStorage.removeItem('report_data' / 'interview_questions' / 'interview_messages')` 清掉前端旧报告与面试快照
- 防御式设计：即使 InterviewPage 的清理失败，这里还会兜底一次。

#### 3. [`ReportPage.tsx`](client/src/pages/ReportPage.tsx) - 重新生成 + 智能缓存
- 新增「重新生成」按钮（顶部栏，`handleRegenerate`），允许用户对当前面试再调一次 AI 评估；
- 缓存策略改为「仅缓存看起来有效的报告」：当 `question_scores` 为空 / 全 0 分 / 无 comment 时不写入 `report_data`，让用户重进时自动重新生成，不会被一次异常输出永久钉住。

#### 4. [`report.py`](ai-service/app/api/report.py) - 强化日志与补齐策略
- 打印 `qa_pairs` 前 600 字预览，方便核对结构化输入是否正确；
- 当 LLM 返回 `question_scores` 数量**少于**题目总数时，按已有顺序保留再补占位（不再无脑覆盖）；
- 异常分支补充打印 `[Report] ❌` 前缀日志，便于在终端快速搜索；
- 完成日志增加 `overall / strengths / weaknesses` 数量，便于一眼判断报告是否健康。

#### 5. [`report_generation.py`](ai-service/app/prompts/report_generation.py) - 压实输出质量
- 新增 6 条**硬性要求**：必须逐题、数组长度等于题数、未作答也要给条目、`comment 至少 20 字`、`strengths/weaknesses/suggestions 至少 15 字`、必须基于实际作答内容；
- 评分标准从 5 档细化（5 分要"具体例子或数据"、3 分"基本切题但空泛"等）；
- JSON 输出示例更详细，明确每个字段的最小内容要求。

### 收益

| 场景 | 修复前 | 修复后 |
| --- | --- | --- |
| 面试结束后立即开始新一场 | 携带上一场的题目和回答 | 后端会话同步清掉，必现拿到新题目 |
| 上游入口（JdInputPage） | 无任何清理逻辑 | 提交新 JD 后主动清前后端缓存 |
| 报告 LLM 输出空 / 全 0 分 | 永远缓存住，用户没法翻身 | 不缓存，且支持「重新生成」一键重试 |
| 排障 | 只能猜 LLM 输出 | 后端打印 `qa_pairs` 预览 + 补齐前后量化日志 |
| LLM 报告质量 | 经常空 comment / 缩水建议 | prompt 强约束最小字数 + 必须基于实际作答 |

### 验证清单
1. 完整答完一场面试，生成报告应**有逐题点评（至少 20 字）**和**优势/改进/建议各 2-3 条**；
2. 后端控制台应看到：
   - `[Report] qa_pairs 预览:` 后跟结构化的题目-回答配对
   - `[Report] 生成完成：question_scores=N 条，overall=X.X，strengths=K，weaknesses=K`
3. 面试结束 → 返回首页 → 上传新简历 / 新 JD 重新面试，AI 应**立刻给出新题目**而非旧题目；
4. 若报告偶现空内容，可点击顶部「重新生成」按钮重新调 AI；
5. （回归）面试结束后 AI 不应再弹出欢迎语；最后一题输入框应正常显示。

### 备注
**会话状态最容易出问题的地方就是"清理时机"**：fire-and-forget 在状态依赖链路上几乎一定会埋雷。本次双 bug 都和清理时机有关——前端跳转太快、上游入口没有兜底——核心修复思路是把清理动作改为同步、并在多个入口冗余清理。报告质量则通过"压实 prompt + 不缓存空报告 + 一键重试"三层兜底解决。

## 🚨 关键修复：AI 面试官自行"脑补全场面试"（面试官+候选人对话被 LLM 一次性说完）

### 现象
从 ReportPage 点击「重新面试」进入新一场，AI 面试官开场就输出了一整段伪造的多轮对话，例如：
```
面试官：好的，我们继续。在'智能基金 Agent 系统'项目中，你是如何设计……
候选人：[假设候选人回答] 在这个项目中，我首先对基金数据进行了……
面试官：了解了。这是最后一题了，感谢你的回答。今天的面试就到这里……
```
候选人还没说话，整场面试就被 LLM "独角戏"完了。

### 根因分析

**根因 A：ReportPage 「重新面试」按钮没有任何清理**
此前 [`ReportPage.tsx`](client/src/pages/ReportPage.tsx) 的「重新面试」只是 `navigate('/interview')`，没清后端 Redis 会话也没清 localStorage。当用户从 ReportPage 直接进 InterviewPage 时，`/session/restore` 有概率恢复到**上一场已结束但残留**的会话（TTL 2h）——LLM 拿到的 `chat_history` 不是空的而是上一场的完整对话，就开始顺着写下去了。

**根因 B：[`chat.py`](ai-service/app/api/chat.py) 把整套 questions_json 都交给 LLM**
之前传参是 `json.dumps(req.questions)` 整个列表。LLM 看到所有题目都摆在面前，很容易"演完"整场面试，尤其在 chat_history 被污染的场景下，会主动补齐"候选人：[假设候选人回答]"来串联剧情。

**根因 C：[`interviewer.py`](ai-service/app/prompts/interviewer.py) prompt 没有明确禁止脑补**
原 prompt 要求"不要帮候选人回答问题"，但没有具体列出反面例子（"候选人："前缀、"[假设候选人回答]"等）。对 LLM 来说，模糊约束等于没约束。

### 改动

#### 1. [`ReportPage.tsx`](client/src/pages/ReportPage.tsx) - 「重新面试」也要清理
- 新增 `handleRetakeInterview` 异步函数，点击「重新面试」时：
  - `await api.delete('/session/clear')` 同步清后端 Redis 会话
  - `localStorage.removeItem` 清 `report_data / interview_questions / interview_messages`
  - 完成后再 `navigate('/interview')`
- 替换原 `onClick={() => navigate('/interview')}` 的直跳方式

#### 2. [`chat.py`](ai-service/app/api/chat.py) - 给 LLM 最小必要上下文
- `questions_json` 从"完整列表"收敛为**摘要对象**：只含 `total / current_index / current_question / next_question / is_last_question`，断绝 LLM 看到后续题目一次性演完的可能
- 新增**脑补安全网**（流式后置过滤）：
  - 维护 `STOP_PATTERNS = ("候选人：", "候选人:", "[假设候选人", "【假设候选人", "假设你回答", "假设候选人回答")`
  - 每次 chunk 到来后扫描全文 buffer，一旦命中禁用模式，把命中点前的部分 flush 出去后立即 `return`，后面全部丢弃
  - 打印 `[Chat] ⚠️ 检测到 LLM 脑补多轮对话，已截断` 便于监控
  - 这是**最后一道防线**：即使 prompt 失效或被越狱，流式截断也能保证前端看不到伪造的"候选人："前缀

#### 3. [`interviewer.py`](ai-service/app/prompts/interviewer.py) - prompt 硬约束
- 新增【最重要的规则】段，明确列出 4 条：
  - 只能输出面试官当前这一刻要说的一段话
  - **严禁**模拟/伪造/脑补/预设候选人的任何回答（含"[假设候选人回答]"等标志性写法）
  - **严禁**一次把后续题目全部说出
  - **严禁**输出多轮"面试官-候选人-面试官-候选人..."对话片段
- 把第 1 题首次触发的流程拆细到 3 步（开场 → 提问 → 立刻结束输出等候选人回答）
- 【严格禁止】列表新增 3 条：候选人/面试官前缀、自行写候选人回答、一次多轮对话

### 收益

| 场景 | 修复前 | 修复后 |
| --- | --- | --- |
| 从 ReportPage 点「重新面试」 | 可能恢复旧 session，LLM 续写上场对话 | 同步清前后端状态，必然拿到新一场 |
| LLM 看到的题目上下文 | 全部 N 道题明文 | 只有当前题 + 下一题摘要 |
| LLM 试图输出"候选人：xxx" | 直接流到前端显示 | 流式截断，用户看不到 |
| prompt 约束 | "不要帮候选人回答"一条模糊要求 | 4 条硬禁 + 3 条补充，有具体反面例子 |

### 验证清单
1. 完整答完一场面试 → 进入 ReportPage → 点击「重新面试」：
   - 新面试页 AI 开场应**只有一句欢迎 + 第 1 题**，不得带任何"候选人："或"[假设候选人回答]"；
   - 若后端出现过截断，控制台应看到 `[Chat] ⚠️ 检测到 LLM 脑补多轮对话，已截断` 日志；
2. 面试结束 → 新上传 JD 再进面试也应表现正常（双重入口都清理）；
3. （回归）面试过程中 AI 不得一次问出 2 道题；最后一题回答完应有结束语。

### 备注
这是典型的「prompt 模糊约束 + 上下文过于充足 + 清理不彻底」三杀：
- **上下文过于充足**让 LLM 有"演完整场"的素材；
- **prompt 模糊**让 LLM 觉得演一下也没关系；
- **清理不彻底**偶尔会把旧对话喂回来点燃整个过程。
修复采用"纵深防御"：前端清理 + 后端收敛输入 + prompt 硬禁 + 流式安全网，任何一层漏了其他层都能兜住。

## 🔧 体验修复：进度条不同步 + 报告 0 题排障 + React 控制台警告

### 现象
1. **进度条停滞**：用户已答完最后一题，左侧"面试进度"仍停在"第 7 题 / 88%"，不会推进到 100%；
2. **报告显示已答 0 题**：明明答完所有题，前端控制台仍报 `[Report] 生成结果看起来异常（全 0 分或无评论），不缓存`，报告页"答题数 0"；
3. **React 警告**：控制台反复出现 `Updating a style property during rerender (animation) when a conflicting property is set (animationDelay) can lead to styling bugs.`。

### 根因分析

**根因 A（进度卡顿）：`setCurrentIndex(nextIndex)` 等到 stream 结束才执行**
[`InterviewPage.streamChat`](client/src/pages/InterviewPage.tsx) 原逻辑是：fetch 流式 → 累计 aiText → 流结束后判定 `aiText.includes('面试结束') || nextIndex > questions.length` → 决定推进或结束。当 LLM 中途说一句"...这是面试结束前的最后一题..."，`includes('面试结束')` 误中，**直接 setInterviewOver(true) 而跳过 setCurrentIndex(nextIndex)**。结果：进度条停在答题前的题号；UI 还会同时进入结束态。

**根因 B（报告 0 题）：判定条件过严 + 缺少诊断信息**
[`report.py`](ai-service/app/api/report.py) 在 `valid_answers === 0` 时直接返回 `{ question_scores: [] }`。前端 [`ReportPage`](client/src/pages/ReportPage.tsx) `looksValid` 判定要求至少一条 `score>0` 或 `comment` 非空——空数组直接 fail。
之前缺乏前端 console / 后端日志，定位时不知道是「localStorage 数据丢了」还是「LLM 生成异常」还是「兜底逻辑没生效」。

**根因 C（React 警告）：CSS shorthand 与 longhand 同时设置**
[`InterviewPage.tsx`](client/src/pages/InterviewPage.tsx) 的 `AudioWave` / `TypingDots`、[`MatchResultPage.tsx`](client/src/pages/MatchResultPage.tsx) 的 sections 列表都同时写了 `animation: 'xxx 1.1s ease-in-out infinite'`（shorthand）和 `animationDelay: '0.1s'`（longhand）。React 18 起对此组合会发警告，因 shorthand 会在重渲染时**覆盖 longhand 的当前值**导致动画跳变。

### 改动

#### 1. [`InterviewPage.tsx`](client/src/pages/InterviewPage.tsx) - streamChat 进度立即推进
- 进入 `streamChat` 第一时间执行 `setCurrentIndex(nextIndex)`（仅当 `nextIndex <= questions.length`），不再等流结束；
- 结束态判定收紧：原 `aiText.includes('面试结束')` 改为：
  - 必须 `nextIndex >= questions.length`（已经在收尾题）
  - **且**关键词出现在文本末尾 80 字内
  - 用正则 `/面试结束|今天的面试.*结束|面试就到这里|本次面试.*结束/`
- 这样中间题如果 LLM 不小心说到"面试结束"等字眼，不会再误触发 setInterviewOver；
- handleEnd 里增加 `console.log('[Interview] 结束面试：questions=X 道，messages=N 条，user 回答=K 条')` 调试输出。

#### 2. [`ReportPage.tsx`](client/src/pages/ReportPage.tsx) - 加详细诊断日志
- `generateReport` 入口打印：`questions / chat_history / user 消息 / 有效作答` 四个计数 + chat_history 头 4 条样本；
- looksValid 失败时把整份 `rawReport` 一并打印，便于贴日志排障；
- 前端这层日志 + 后端 `[Report] qa_pairs 预览` + `[Report] 生成完成` 三处对照，可以精确定位是哪一层丢了数据。

#### 3. CSS shorthand 合并（消除 React 警告）
- [`InterviewPage.tsx`](client/src/pages/InterviewPage.tsx) `AudioWave`：`animation: \`wave 1.1s ease-in-out ${i * 0.1}s infinite\`` 把 delay 直接写进 shorthand；
- [`InterviewPage.tsx`](client/src/pages/InterviewPage.tsx) `TypingDots`：同上，`animation: \`blink 1.4s ease-in-out ${i * 0.22}s infinite\``；
- [`MatchResultPage.tsx`](client/src/pages/MatchResultPage.tsx) 第 207 行：sections 列表渲染处合并 delay 进 `animation: \`msgIn 0.4s cubic-bezier(...) ${0.3 + si * 0.12}s both\``；
- 原则：**永远不要在同一个元素 style 上同时写 `animation` 和 `animationDelay/animationDuration` 等 longhand**。

### 收益

| 现象 | 修复前 | 修复后 |
| --- | --- | --- |
| 答完最后一题进度条 | 停在 88% / 87.5%（最后一题没推进） | 立即跳到 100% |
| 中间题 AI 偶尔说到"结束" | 误触发 setInterviewOver | 仅"末尾 80 字 + 末题"才生效 |
| 报告 0 题 | 没有诊断信息 | 前端 console 直接看到各计数 + 原始报告 |
| React 警告 | 控制台刷屏 | 干净 |

### 验证清单
1. 完整答完一场面试，左侧进度应**最终到 100%**，状态切到"面试已结束"；
2. 中间答题时 AI 即使说出"...这不是面试结束的题"也不会被误判结束；
3. 控制台进入 ReportPage 应看到 `[Report] 准备生成：questions=N 道，chat_history=M 条，user 消息=K 条，有效作答=V 条`；若 V===0，去检查 InterviewPage 控制台的 `[Interview] 结束面试：...`，看 messages 是否完整；
4. 控制台应**不再出现** `Updating a style property during rerender (animation)` 警告。

### 备注
本次修复的核心思路是**「优先暴露问题再彻底修复」**：
- 进度卡顿是简单状态机 bug，先把推进时机前置；
- 报告 0 题暂时没有 100% 复现，所以先把诊断日志加全（前端+后端），下次复现可一键定位；
- React 警告是 CSS API 易错点，统一用 shorthand 一种写法收敛。

## 🔧 综合修复：4 大 bug（追问跨题、报告空数据、AI 跳题、状态机时序）

### 现象
1. **AI 实际问的题与左侧进度题不一致**：跨题、跳过中间题、回头复述前面的题
2. **题目顺序乱**：questions 数组本身顺序正确（RAG 5 道按 score 排序 + LLM 3 道），但 AI 没按 currentIndex 严格问
3. **答完第 7 题就显示"面试结束"**，跳过第 8 题
4. **报告页面空白**：`/report/generate` 请求负载里 `questions: []`、`chat_history: []`，但同一时间 `/session/save` 请求体里 messages 是完整的

### 根因（4 个独立但相关的问题）

| Bug | 根因 |
|---|---|
| AI 跳题 | interviewer prompt 给了 LLM 太大自主权，让它"自己挑题问" |
| 顺序乱 | 同上，LLM 自由发挥导致表现像顺序乱 |
| 第 7 题就结束 | 追问轮 `streamChat(newMessages, currentIndex+1)` 把 nextIndex 推成 8（=questions.length），命中"末题+末尾80字含'结束'"误判 |
| 报告空数据 | handleEnd 在跳转前 `await /session/clear` 把 Redis 清了；用户点"重新生成"或刷新 ReportPage 时 localStorage 被某处清掉，没有任何兜底数据源 |

### 修复方案（纵深四层防御）

#### 1. AI 跳题/跨题/回头 → interviewer prompt 硬约束（[`ai-service/app/prompts/interviewer.py`](ai-service/app/prompts/interviewer.py)）
- 重写 prompt，把【🔥 最高优先级硬性规则】放最前
- 4 条核心硬约束：
  1. **禁止跳题**：你只能问 system 注入的【当前题目】`{current_question}`
  2. **禁止跨题/回头**：禁止从 questions_json 摘要里挑别的题、禁止回头复述前题
  3. **追问 vs 推进必须严格按 `need_follow_up` 字段**：true→只问追问问题、不进下一题；false→直接说当前题
  4. **不要自作主张结束面试**：仅当 `current_index == total_count` 且 `need_follow_up=false` 才允许说"面试结束"
- 把 questions_json 改名为"题目摘要（仅供参考，不要据此提问）"明确语义

#### 2. 追问轮被误判结束 → streamChat 增加 isFollowUpRound 参数（[`client/src/pages/InterviewPage.tsx`](client/src/pages/InterviewPage.tsx)）
- streamChat 新增第 4 参数 `isFollowUpRound: boolean = false`
- 追问轮：
  - 不调用 `setCurrentIndex(nextIndex)`（进度不推进）
  - 流式结束直接 return，**完全跳过结束态判定**
- 正常推进轮：保留原逻辑（进度立即推进 + 末尾 80 字 + 末题 + 正则）
- 这样即使追问的 nextIndex 被传成 8，也不会触发结束判定

#### 3. autoSave 状态时序 bug → 显式状态 override（[`InterviewPage.tsx`](client/src/pages/InterviewPage.tsx)）
**根因**：handleSend 追问分支里 `setIsFollowUp(true)` 后**立刻** `autoSave()`，autoSave 读到的 `isFollowUp` state 还是旧值 `false`，Redis 里写入的就是 `is_follow_up: false`
- 重构 `autoSave(override?)` 接收显式状态对象
- handleSend 三处调用（追问/正常推进/上一轮追问后切下题）+ handleSkip 都改为传入显式 override
- 修复后：刷新恢复 Redis 时拿到的 `isFollowUp` 状态准确，状态机不会错乱

#### 4. 报告 0 题/空数据 → ReportPage 增加 /session/restore 兜底 + handleEnd 移除 /session/clear
- [`InterviewPage.tsx`](client/src/pages/InterviewPage.tsx) `handleEnd`：
  - **移除** `await api.delete('/session/clear')`，让 Redis 在结束后还保留本场会话
  - 增加同步 `await autoSave({...})` 把最新 messages/questions 一并写入 Redis 作兜底
  - 后端 session 清理统一交给 2 处：① JdInputPage 提交新 JD（开始下一场）② ReportPage handleRetakeInterview（"重新面试"按钮）
- [`ReportPage.tsx`](client/src/pages/ReportPage.tsx) `generateReport`：
  - 优先读 localStorage（保持原行为）
  - 如果 `questions` 或 `chat_history` 为空，**自动调 `/session/restore`** 从 Redis 拉真实数据回填
  - 兜底成功的同时把 Redis 数据写回 localStorage，下次直接命中
  - 这样无论用户从什么路径进 ReportPage（handleEnd 跳转 / 刷新 / 直接访问 URL），只要 Redis 还有会话就拿得到数据

### 关键代码改动总结

| 文件 | 修改 |
|---|---|
| [`ai-service/app/prompts/interviewer.py`](ai-service/app/prompts/interviewer.py) | 完整重写：4 条最高优先级硬性规则 + 5 条行为细则 + 6 条禁止项 |
| [`client/src/pages/InterviewPage.tsx`](client/src/pages/InterviewPage.tsx) | streamChat 增加 isFollowUpRound 参数；autoSave 增加 override；handleSend/handleSkip/handleEnd 三处适配；handleEnd 移除 /session/clear |
| [`client/src/pages/ReportPage.tsx`](client/src/pages/ReportPage.tsx) | generateReport 增加 /session/restore 兜底分支 |

### 验证步骤
1. 重新跑一次完整 8 题面试
2. 观察 AI 是否严格按第 1→2→3...→8 题顺序问，不跳题、不回头、不跨题
3. 观察追问后是否安全推进到下一题、第 8 题是否被完整问到
4. 答完第 8 题进入报告页，应能看到完整 8 题逐题评估
5. 在报告页刷新，仍能看到完整报告（来自 localStorage 缓存）
6. 在报告页点"重新生成"，应能从 /session/restore 拉到原始数据再次生成（不会出现空报告）

### 备注
这次"纵深四层"的核心思想：
- **prompt 层**（约束 LLM 行为）+ **状态机层**（前端正确推进）+ **数据层**（多源兜底）+ **时序层**（autoSave 显式状态）
- 任何一层失效，下一层兜住，不再出现"全链路失败导致用户看到空白"的情况
